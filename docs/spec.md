# SKU別販売予測アプリ 仕様書（Claude Code実装用）

**作成日**: 2026年6月19日
**対象部門**: コマーシャルエクセレンス部門
**対象製品**: ジェネリック医薬品・バイオシミラー

---

## 1. プロジェクト概要

### 1.1 目的
ジェネリック医薬品・バイオシミラーのポートフォリオについて、**SKU単位**で**今後36ヶ月の月次販売予測**を生成するアプリケーションを構築する。予測には以下を組み込む：

- 当該SKUが属する**疾患領域・成分全体の市場トレンド**（IQVIAデータ）
- 当該SKUが属する成分の**ジェネリック/バイオシミラー浸透率**の推移とその将来カーブ
- 薬価改定・後発品使用促進策など**制度変更の影響**をパラメータとして調整可能にする
- 金額は**日本円(JPY)／米ドル(USD)**のいずれでも表示でき、レートは調整可能
- **制度変更の影響を加味したシナリオ**と**加味しないシナリオ**を分けて比較できる
- 製品単位の予測をSKUへ配分する際は、SKU構成比の**トレンド（変化傾向）**を考慮する
- 経営層向けの**PowerPoint出力**
- 予測ロジックを口頭で説明するための**トーク原稿（用語解説付き）**の出力

### 1.2 想定ユーザー
- コマーシャルエクセレンス部門のアナリスト（自分自身）
- 営業企画・マーケティング部門（閲覧・シナリオ確認用途）
- 経営層（PowerPointレポートの閲覧）

### 1.3 利用シーン
- 月次/四半期の事業計画見直し
- 薬価改定（2年に1度の通常改定、毎年の中間年改定）発表時の予測再計算
- 新規競合（後発品・バイオシミラー）参入時のシナリオ分析
- 経営層向けレポーティング（PowerPoint配布）
- グローバル本社向け報告（USD換算が必要な場合）

## 2. 業界特性に関する前提（ジェネリック/バイオシミラー固有）

予測ロジックを設計する上で、以下の業界特性を反映する必要がある。

| 特性 | 内容 | 予測モデルへの反映 |
|---|---|---|
| 浸透率カーブ | 新規参入後、S字カーブ（ロジスティック曲線）で浸透率が上昇し、その後飽和する | ロジスティック関数 or Bass拡散モデルでカーブフィッティング |
| 薬価改定 | 原則2年に1度（中間年は対象品目のみ）薬価が引き下げられる。改定年月・改定率が既知の場合はSell-in/Sell-out額に反映 | 改定月をイベントフラグとして時系列に挿入し、価格弾力性パラメータで調整。**「制度変更を加味するか否か」をシナリオの主軸として分離**（詳細は4.1, 6章） |
| 後発品使用促進策 | 政府目標（後発品使用割合目標）の変更が浸透率の伸びに影響 | 浸透率上限（飽和点）パラメータとして調整可能に |
| バイオシミラー特性 | 浸透が低速・診療科や施設単位での切替が中心。スイッチング（既存患者の切替）と新規処方の区別が必要 | 新規患者数 × 浸透率 ＋ 既存患者切替率 のモデルに分離 |
| Sell-in/Sell-outのギャップ | 流通在庫変動により短期的にSell-inとSell-outがズレる（薬価改定前の特需・改定後の調整等） | **卸別月末在庫データ**を用いてSell-in = Sell-out + 在庫増減として直接分解。推定ではなく実測値で補正可能 |
| 疾患領域の市場成長 | 高齢化・診断率向上等によるオーガニック市場成長と、自社製品の市場内シェアを分離して捉える | 「疾患領域市場サイズ予測 × 自社製品の市場内シェア予測」の二段階構造 |
| SKUの多様性 | 同一製品でも規格・包装単位（PTPシート数、バイアル本数等）が異なる複数SKUが存在し、卸の発注単位もSKU単位 | 市場トレンド・浸透率は**成分単位**で算出し、各SKUへは**SKU構成比（ミックス比率）**で配分する2階層構造とする |

## 3. データの粒度とマスタ階層

### 3.1 階層構造
本アプリでは、データの粒度を以下の4階層で管理する。

```
疾患領域 (Therapeutic Area)
  └─ 成分 (Ingredient / 一般名)         ← IQVIA市場データ・浸透率はこの単位で集計
       └─ 製品 (Product / 自社製品名)    ← ブランドとしての括り
            └─ SKU (規格・包装単位)      ← 予測の最終出力単位、卸の発注・在庫もこの単位
```

- **市場トレンド・ジェネリック浸透率**は「成分」単位で算出する（IQVIAデータの提供粒度、および同一成分内での後発品競合状況を正しく捉えるため）
- **Sell-in／Sell-out／在庫**は「SKU」単位で保持する（卸との実際の取引単位に合わせるため）
- 自社製品予測の最終出力は **SKU単位**とし、必要に応じて製品単位・成分単位・疾患領域単位に集計（ロールアップ）して表示できるようにする
- **SKU構成比（ミックス比率）**：同一製品内の各SKUが、製品全体の販売量に占める割合。過去の実績から**時系列のトレンド**（増加/減少傾向、季節性等）を捉えてフィッティングし、将来予測ではこのトレンドを延長した比率を用いて製品レベルの予測をSKUに配分する（単純な過去平均の固定値は使わない。詳細は6.1節Step4）

### 3.2 SKUマスタの主要属性
| フィールド | 内容 |
|---|---|
| sku_id | SKU識別子（自社の品目コード等） |
| product_id | 紐づく製品ID |
| ingredient_id | 紐づく成分ID |
| therapeutic_area_id | 紐づく疾患領域ID |
| package_form | 包装形態（PTP/バラ/バイアル等） |
| package_size | 包装単位数量（例：100錠/シート、10mL/バイアル） |
| strength | 規格・含量 |
| jan_code | JANコード等の外部識別子（あれば） |
| type | generic／biosimilar |
| launch_date | 自社上市日 |

## 4. データソース定義

### 4.1 IQVIA市場データ
| 項目 | 内容 |
|---|---|
| 粒度 | 成分別・月次・(可能であれば)都道府県/地域別 |
| 主要フィールド | 疾患領域(ATCコード等)、成分名(一般名)、製品名、メーカー、剤形、規格、販売額、販売数量(錠/バイアル等)、薬価 |
| 用途 | ①疾患領域全体の市場規模トレンド ②同一成分内での自社製品シェア・ジェネリック浸透率算出 |
| 取込形式 | CSV/Excel（IQVIAから定期エクスポートされるものを想定。フォーマットは現物確認の上で調整） |
| 更新頻度 | 月次（IQVIAの提供サイクルに依存、通常1〜2ヶ月のラグ） |
| 通貨 | 原則JPYで提供される前提（他通貨の場合は取込時にJPY換算） |

### 4.2 自社Sell-inデータ
| 項目 | 内容 |
|---|---|
| 粒度 | **SKU別**・月次・(可能なら)得意先/卸別 |
| 主要フィールド | SKUコード、得意先(卸)、出荷数量、出荷額(JPY)、出荷日 |
| 用途 | 実績の起点データ、在庫変動補正の基礎 |

### 4.3 自社Sell-outデータ
| 項目 | 内容 |
|---|---|
| 粒度 | **SKU別**・月次・(可能なら)施設区分別 |
| 主要フィールド | SKUコード、卸からの出庫数量/額(JPY)、納品先区分 |
| 用途 | 真の需要シグナルとして予測のベースラインに使用 |

### 4.4 卸別月末在庫データ
| 項目 | 内容 |
|---|---|
| 粒度 | **SKU別**・月次・卸別 |
| 主要フィールド | SKUコード、卸名、月末在庫数量、(可能なら)月末在庫金額(JPY) |
| 用途 | Sell-in/Sell-outギャップの**実測補正**、在庫日数(Days of Inventory)の算出、薬価改定前後の特需・在庫調整の検知 |
| 取込形式 | CSV/Excel（卸別・月次集計を想定） |
| 更新頻度 | 月次 |

> このデータにより、Sell-in/Sell-outギャップ補正は推定ではなく**実測ベースの直接計算**が可能になる。ある月の卸在庫増減 = 当月Sell-in − 当月Sell-out という恒等式を用いて、薬価改定前の積み増しや改定後の調整等を精緻に説明・予測する。

### 4.5 制度・イベントマスタ
薬価改定スケジュール、後発品使用促進策の目標値変更、NHI関連の制度変更を管理するテーブル。

| フィールド | 内容 |
|---|---|
| event_date | 発生月 |
| event_type | 薬価改定／中間年改定／使用促進策目標変更／その他 |
| impact_scope | 全製品／特定疾患領域／特定成分／特定SKU |
| impact_value | 価格変化率、浸透率上限変化など（パラメータとして編集可能） |
| memo | 自由記述 |

### 4.6 為替レートマスタ（新規追加）
| フィールド | 内容 |
|---|---|
| rate_type | 実績(historical) ／ 予測仮定(forecast assumption) |
| period | 年月（実績の場合）／適用開始月（予測仮定の場合） |
| jpy_per_usd | 1USD当たりのJPYレート |
| updated_by, updated_at | 変更履歴用 |

> 予測仮定レートはユーザーが画面上で自由に編集可能。デフォルトは直近実績レートを将来に固定するシンプルな仮定とし、必要に応じて期間ごとに変化させる設定（例：緩やかな円安/円高トレンド）も可能にする。

## 5. 出力要件

### 5.1 予測出力
- **SKUごとに今後36ヶ月**の月次予測（Sell-in額/数量、Sell-out額/数量）
- 金額は**JPY／USDの表示切替**が可能（為替レートはユーザー調整可能なパラメータを適用）
- 信頼区間（例：80%/95%）を併記
- **製品単位・成分単位・疾患領域単位**へのロールアップ表示（SKUを合算）
- シナリオは以下の2軸で構成し、それぞれ比較表示できる（詳細は6章）
  - 軸1：**制度変更影響の加味（ON）／非加味（OFF）**
  - 軸2：ベース／楽観／悲観（軸1の各設定内でのさらなる幅）

### 5.2 関連性の可視化
SKU（または製品・成分単位にロールアップ）ごとに以下を1画面で確認できるダッシュボード：

1. **疾患領域・成分市場トレンド**：当該成分が属する市場規模（実績＋予測）の推移グラフ
2. **ジェネリック/バイオシミラー浸透率**：当該成分市場における浸透率の実績推移＋将来カーブ
3. **自社SKU/製品の予測**：上記2つに対する自社の予測販売額/数量（積み上げ or オーバーレイ表示）
4. **要因分解（ウォーターフォール）**：前年同月比の予測変化を「市場成長要因／浸透率変化要因／薬価改定要因／その他」に分解
5. **流通在庫の状況**：卸別在庫日数(Days of Inventory)の推移、Sell-in/Sell-outとの差分（在庫増減）を可視化
6. **制度変更影響の比較**：「加味シナリオ」と「非加味シナリオ」を並べた予測比較（差分＝制度変更による影響額として明示）

### 5.3 エクスポート
- Excel(xlsx)出力（予測テーブル＋グラフ、JPY/USD両方の列を含める）
- **経営報告用PowerPoint(pptx)出力**（詳細は5.4）
- **ロジック解説用トーク原稿の出力**（PowerPointとは別ファイル。詳細は5.5）

### 5.4 PowerPoint出力の要件
経営層向けレポートとして、以下を含む数枚構成のPowerPointを自動生成する。

| スライド | 内容 |
|---|---|
| 1. サマリー | 対象期間・対象製品（またはポートフォリオ全体）の予測総額、前年比、主要トピック（制度変更等）の箇条書き |
| 2. 疾患領域・成分トレンド | 市場規模推移と浸透率カーブのグラフ（製品/成分単位） |
| 3. 自社販売予測 | 36ヶ月予測グラフ（実績＋予測、信頼区間付き）、JPY/USD切替後の数値を反映 |
| 4. 制度変更影響比較 | 「加味シナリオ」vs「非加味シナリオ」の比較グラフと差分サマリー |
| 5. 要因分解 | ウォーターフォールチャート（市場成長／浸透率／薬価改定／その他） |
| 6. リスク・アクション | 在庫状況・要確認事項（メモ欄、手動編集可） |

- 出力時にユーザーが対象製品（または製品グループ／全体ポートフォリオ）、対象シナリオ、通貨を選択できるようにする
- 自社のブランドカラー・フォントを反映したテンプレートを用意（将来的にテンプレートファイルを差し替え可能な設計とする）

### 5.5 ロジック解説用トーク原稿の出力（新規追加）
PowerPointのグラフだけでは伝わりにくい「なぜこの予測になったのか」を、口頭説明できる形の原稿として別ファイルで出力する。経営層・他部門への説明時にそのまま読み上げ／参考にできることを目的とする。

| 項目 | 内容 |
|---|---|
| 出力形式 | Word(docx)（またはMarkdown／画面上での表示も可。詳細は実装時に確認） |
| 想定読み手 | 説明者自身（自分）の手元原稿、または説明先（経営層・他部門）への配布資料 |
| 構成 | ①予測の目的・対象（製品/SKU、期間、シナリオ）のサマリー文 ②疾患領域・成分市場トレンドの解説文 ③ジェネリック/バイオシミラー浸透率カーブの解説文（なぜS字カーブになるのか、現在地点の説明） ④SKU配分の考え方（SKU構成比のトレンドをどう反映したか） ⑤制度変更「加味」vs「非加味」シナリオの違いと、その差分が意味すること ⑥要因分解（市場成長／浸透率／薬価改定／その他）の解説文 ⑦想定される質疑への回答例（Q&A形式、3〜5問程度） |
| 文章スタイル | 口語的・平易な説明文（プレゼンでそのまま話せる文体）。社内向けの実務的な言い回しを基本とする |
| 専門用語の解説 | 巻末に**用語集（グロッサリー）**を付け、本文中で使用した専門用語を平易に解説する。対象語彙の例：ロジスティック曲線（S字カーブ）、ジェネリック浸透率、Sell-in／Sell-out、在庫日数(Days of Inventory)、信頼区間、薬価改定／中間年改定、MAPE等の予測精度指標、シナリオ（加味/非加味）等 |
| 生成方法 | 各製品/SKU・シナリオの実際の予測数値をテンプレート文に埋め込んで自動生成する。文章テンプレートはセクション単位で部品化し、数値に応じて表現（増加/減少、強気/弱気等）を切り替えるロジックを実装する。より自然な文章にしたい場合は、生成した数値入りのドラフトをAnthropic API（Claude）に渡して文章を整える仕上げ工程を追加することも可能（オプション） |
| パラメータ | 対象SKU/製品/疾患領域、対象シナリオ、詳細度（簡易版／詳細版）をユーザーが選択して生成 |

## 6. 予測ロジック（手法設計）

### 6.1 全体構造（2段階モデル、SKUへの配分を含む）

```
Step 1: 疾患領域・成分市場サイズ予測
   IQVIA市場データ（成分単位・時系列） → 時系列モデルで将来トレンドを予測
   （線形トレンド＋季節性、必要に応じてARIMA/Prophet等）

Step 2: ジェネリック/バイオシミラー浸透率予測（成分単位）
   浸透率の実績推移 → ロジスティック曲線（S字カーブ）でフィッティング
   パラメータ：浸透率上限(飽和点)、立ち上がり速度、変曲点タイミング
   → 制度変更イベント（使用促進策強化等）で上限・速度を調整可能

Step 3: 自社製品シェア予測（成分単位 → 製品単位）
   自社製品の「ジェネリック/バイオシミラー内シェア」の推移
   → 競合動向（新規参入時期・想定シェア）をパラメータとして反映

Step 4: SKUへの配分（SKU構成比のトレンドを考慮）
   各SKUの過去のSKU構成比（製品全体に対する比率）の時系列に対し、
   トレンド推定（線形回帰・指数平滑・または緩やかな変化を想定したロジスティック的な
   遷移モデル等）を適用し、将来36ヶ月分の構成比を予測する
   → 新規SKU投入や包装変更により構成比が急変している場合は、変化点を考慮した
     区間ごとのトレンド推定（変化点検知＋区間別フィッティング）を行う
   → 各月において全SKUの構成比合計が1になるよう正規化（正規化前の予測値の比率で配分）
   製品単位の予測 × 上記の将来構成比（月次） → SKU単位の予測値に変換
   構成比のトレンド・変化点は手動調整可能なパラメータとしても提供する（8章）

Step 5: 統合（シナリオ軸の適用）
   自社販売予測(SKU) = [Step1×Step2×Step3×Step4の基礎予測]
               × 薬価改定による価格調整係数 ※「制度変更加味シナリオ」のみ適用
               ± 卸別在庫変動補正（Sell-in用、実測在庫データに基づく）
   → 「制度変更非加味シナリオ」では薬価改定・制度イベントの調整係数を1.0（影響なし）として計算し、
      オーガニックなトレンドのみの予測を別途出力する
```

### 6.2 Sell-in / Sell-outの扱い（卸別月末在庫データを活用）
- **Sell-out**をベース需要として上記モデルで予測
- **在庫変動の実測**：卸別月末在庫データから、当月の在庫増減 = 当月Sell-in − 当月Sell-outが恒等式として成立することを利用し、過去の在庫増減パターンを実測ベースで分析
- **在庫日数(Days of Inventory)の算出**：月末在庫 ÷ 直近のSell-out（または出庫ペース）から、卸ごとの適正在庫水準・逸脱を検知
- **Sell-in予測**：Sell-out予測 ± 在庫変動補正項で算出。補正項は以下の2つの成分に分解する
  - ①**定常的な在庫変動**：通常の在庫日数の範囲内の変動（季節性・卸の発注サイクル等）
  - ②**イベント起因の在庫変動**：薬価改定前の積み増し購買、改定後の在庫調整等。`regulatory_events`マスタに紐づけてパラメータ化（「制度変更加味シナリオ」のみ反映）
- 在庫日数が異常値を示す月は、Sell-in実績からSell-out推定への変換時に除外またはフラグ付けし、モデルの誤学習を防止

### 6.3 通貨換算の扱い
- 全ての実績データ・予測計算はJPYベースで内部保持する（IQVIA・Sell-in・Sell-out・在庫データは原則JPY）
- 表示時にユーザーが選択した通貨（JPY/USD）と為替レートパラメータ（4.6節）に基づき、表示直前に換算する
- USD表示時は、為替レート自体の変動も将来予測の不確実性要素として注記する（レート変更による感度を簡易的に確認できるよう、レートを変えた場合の再計算を即時反映）

### 6.4 検証方法
- バックテスト：直近12ヶ月を予測期間として除外し、実績と比較（MAPE、RMSE等で精度評価）
- 主要SKU・製品については月次で予測精度をモニタリングし、モデル再学習のトリガーとする

## 7. シナリオ設計

シナリオは以下の2軸の組み合わせで構成する。

### 7.1 軸1：制度変更影響の加味／非加味
| シナリオ | 内容 | 用途 |
|---|---|---|
| **加味シナリオ（Regulatory-Adjusted）** | 既知・想定される薬価改定、後発品使用促進策の変更等を予測に反映 | 実務上の事業計画・予算策定のベース |
| **非加味シナリオ（Organic / Baseline）** | 制度変更要因を一切反映せず、市場トレンド・浸透率の自然な推移のみで予測 | 制度変更そのものの影響額を切り出して把握する／オーガニックな事業成長力を評価する |

両シナリオの差分（加味シナリオ − 非加味シナリオ）を「制度変更による影響額」として自動算出し、ダッシュボード・PowerPointの両方で明示する。

### 7.2 軸2：楽観／ベース／悲観
各軸1のシナリオ内で、主要パラメータ（浸透率上限、市場成長率、競合参入タイミング等）に楽観的/悲観的な値を設定した派生シナリオを持つ。

| シナリオ名 | 例 |
|---|---|
| ベース | デフォルトパラメータ |
| 楽観 | 浸透率上限+α、市場成長率+α、競合参入後ろ倒し |
| 悲観 | 浸透率上限-α、市場成長率-α、競合参入前倒し |

### 7.3 シナリオ管理
- `scenario_name`は「regulatory_adjusted_base」「regulatory_excluded_base」「regulatory_adjusted_optimistic」等、2軸を組み合わせた識別子で管理
- ダッシュボード・PowerPointでは、軸1（加味/非加味）の比較をデフォルト表示とし、軸2（楽観/悲観）は任意で追加表示できるようにする

## 8. パラメータ調整機能（制度変更対応）

ユーザーが画面上で調整可能なパラメータ一覧：

| カテゴリ | パラメータ | デフォルト値の出典 | 調整理由の例 |
|---|---|---|---|
| 薬価 | 次回改定予定月 | 制度カレンダーから自動設定 | 改定スケジュール変更 |
| 薬価 | 改定率（成分別/疾患領域別） | 過去改定率の平均 or 直近改定率 | 改定率予測の変更 |
| 浸透率 | 浸透率上限（飽和点） | 過去類似品の実績平均 | 後発品使用促進策強化/緩和 |
| 浸透率 | 立ち上がり速度 | カーブフィッティング結果 | 切替キャンペーン等の施策効果 |
| 浸透率 | 変曲点タイミング | カーブフィッティング結果 | 新規競合参入予測の前倒し/後ろ倒し |
| 市場 | 疾患領域市場成長率 | IQVIA過去トレンドの外挿 | 診断率向上・高齢化等のシナリオ変更 |
| 競合 | 新規競合参入時期・想定シェア | 手動入力 | 競合情報のアップデート |
| 在庫 | 適正在庫日数（卸別/全体） | 実測在庫データの過去レンジから算出 | 流通施策変更、卸との取引条件変更 |
| 在庫 | イベント起因の在庫変動係数（改定前積み増し率/改定後調整率） | 過去の薬価改定時の実測在庫変動から算出 | 次回改定の規模・タイミング変更 |
| SKUミックス | SKU構成比のトレンド係数（傾き）／変化点 | 過去実績のトレンドフィッティング結果 | 新規SKU投入、包装変更、卸の発注単位変更等 |
| 通貨 | 為替レート（JPY/USD、期間別） | 直近実績レート | 為替前提の変更、本社報告用レート指定 |
| シナリオ | 楽観/悲観の調整幅（軸2） | - | 経営報告用シナリオ分岐 |
| レポート | トーク原稿の詳細度（簡易/詳細） | 詳細版をデフォルト | 説明先・想定時間に応じた調整 |

> パラメータ変更はSKU単位・製品単位・成分単位・疾患領域単位のいずれでも適用できるようにし、変更履歴（誰が・いつ・何を変更したか）をログとして保持する。

## 9. システムアーキテクチャ（推奨構成）

開発はClaude Codeで進めることを前提に、以下の構成を推奨する（既存のSalesforceパイプラインダッシュボード等での実績を踏まえ、React+Python APIの構成と親和性が高い）。

```
[データ取込層]
  CSV/Excel(IQVIA, Sell-in, Sell-out, 卸別月末在庫, 為替レート) → pandas による前処理・正規化
  → ローカルDB(SQLite想定、将来的にPostgreSQLへ拡張可)

[予測エンジン層] (Python)
  - pandas / numpy: データ整形
  - statsmodels or Prophet: 時系列予測(疾患領域・成分市場サイズ)
  - scipy.optimize: ロジスティック曲線フィッティング(浸透率)
  - FastAPI: 予測ロジックをAPI化し、パラメータ変更を受けて再計算
  - 通貨換算ロジック：内部はJPY保持、表示/出力時にレートパラメータで変換

[フロントエンド層] (React + Tailwind)
  - SKU/製品/成分の選択（階層ナビゲーション）、パラメータ調整パネル
  - JPY/USD表示切替トグル、為替レート入力
  - グラフ表示: Recharts（市場トレンド、浸透率カーブ、予測、ウォーターフォール、シナリオ比較）
  - シナリオ比較ビュー（軸1×軸2）
  - Excelエクスポート機能(SheetJS)

[レポート生成層] (Python)
  - python-pptx: 経営報告用PowerPoint自動生成
  - python-docx: ロジック解説用トーク原稿の自動生成（テンプレート文＋数値埋め込み、用語集を巻末に付与）
  - グラフ画像はmatplotlib等でレンダリングしスライドに埋め込み、またはフロントエンドのグラフをエクスポートして埋め込み
  - (オプション) トーク原稿の文章仕上げにAnthropic API(Claude)を利用する場合は、生成済みドラフトをプロンプトとして渡し、文章の自然さのみを整える(数値・結論は変更しない)

[データ永続化]
  - パラメータ設定・シナリオ・変更履歴・為替レート設定をDBに保存
  - 将来的にHubSpot等との連携は不要（社内分析専用ツールのため）
```

> 個人利用・小規模チーム利用が前提であれば、SQLite + ローカルFastAPI + Reactのシンプル構成で十分。将来的に複数ユーザーでの共有が必要になった場合は、PostgreSQL + 認証機能の追加を検討。

## 10. データモデル（概要）

```
therapeutic_areas（疾患領域マスタ）
  - therapeutic_area_id, name

ingredients（成分マスタ）
  - ingredient_id, therapeutic_area_id, name(一般名)

products（製品マスタ）
  - product_id, ingredient_id, product_name, type(generic/biosimilar)

skus（SKUマスタ）
  - sku_id, product_id, package_form, package_size, strength, 
    jan_code, launch_date

market_data（IQVIA由来、成分単位）
  - ingredient_id, period(年月), total_market_sales_jpy, 
    total_market_units, generic_penetration_rate

sellin_data（SKU単位）
  - sku_id, period, distributor, quantity, amount_jpy

sellout_data（SKU単位）
  - sku_id, period, facility_type, quantity, amount_jpy

inventory_data（卸別月末在庫、SKU単位）
  - sku_id, period, distributor, ending_inventory_qty, 
    ending_inventory_amount_jpy(任意), days_of_inventory(算出値)

sku_mix_ratio（SKU構成比、トレンド対応）
  - sku_id, period, actual_mix_ratio(実績値), 
    trend_model_type(linear/exp_smoothing/piecewise等), 
    trend_params(係数、変化点情報等), forecast_mix_ratio(予測値)

glossary（専門用語集、トーク原稿用）
  - term, plain_explanation, category(例：予測手法/業界用語/指標)

regulatory_events（制度マスタ）
  - event_id, event_date, event_type, impact_scope, impact_value, memo

fx_rates（為替レートマスタ）
  - rate_type(historical/forecast_assumption), period, jpy_per_usd, 
    updated_by, updated_at

forecast_parameters（パラメータ設定）
  - target_id(sku_id/product_id/ingredient_id/therapeutic_area_id), target_level,
    parameter_name, value, scenario_axis1(adjusted/excluded), 
    scenario_axis2(base/optimistic/pessimistic), updated_by, updated_at

forecast_results
  - sku_id, period, scenario_axis1, scenario_axis2, 
    sellin_forecast_jpy, sellout_forecast_jpy,
    confidence_lower, confidence_upper, generated_at
```

## 12. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | SKU数百〜千件規模でも、パラメータ変更後の再計算は数秒以内に画面反映 |
| データ保護 | 自社販売データのため、社外への送信・共有なし（ローカル環境での動作を基本とする） |
| 拡張性 | SKU/製品/成分/疾患領域マスタの追加が容易な設計（ハードコード回避） |
| 監査性 | パラメータ変更履歴・為替レート変更履歴を保持し、予測値の再現性を担保 |
| 保守性 | 予測ロジック（Step1〜5）は関数単位で分離し、手法差し替え（例：Prophet→別モデル）が容易な構造にする |
| 国際化対応 | 通貨表示はJPY/USDの2種類を基本とし、将来的に他通貨追加が容易な設計（レートテーブルを通貨コード別に拡張可能） |

## 13. 開発ステップ（推奨マイルストーン）

1. **データ基盤構築**：IQVIA/Sell-in/Sell-out/卸別月末在庫のサンプルデータでDB・前処理パイプラインを構築（SKU/製品/成分/疾患領域の階層マスタを含む）
2. **予測エンジン（疾患領域・成分市場予測）**：単一成分でのプロトタイプ作成、バックテストで精度確認
3. **浸透率モデル**：ロジスティック曲線フィッティングの実装、制度イベントの反映ロジック追加
4. **統合予測（製品→SKU配分）**：Step1〜4の統合。SKU構成比のトレンド・変化点フィッティングを実装し、SKU単位での36ヶ月予測の出力確認
5. **シナリオ軸の実装**：制度変更加味/非加味の2軸、楽観/悲観の派生シナリオ
6. **通貨換算機能**：為替レートパラメータの実装、JPY/USD表示切替
7. **フロントエンド構築**：ダッシュボード・パラメータ調整UI・グラフ表示
8. **エクスポート機能**：Excel出力、PowerPoint自動生成（python-pptx）、ロジック解説トーク原稿の自動生成（python-docx、用語集付き）
9. **複数SKU・製品への展開・運用化**：マスタ拡充、月次運用フローの確立

- 利用者数（自分one personか、チーム共有か）→ 認証・マルチユーザー対応の必要性
- 既存のSalesforce/Qlik/Alteryx環境との連携要否（データ取込を既存Alteryxワークフローから受け取る等の可能性）
